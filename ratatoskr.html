Ratatoskr: Compositional Tool Retrieval at Organizational Scale

Harshavardhanan Deekeswar · April 2026

1. The Problem

Inside any large organization today, agents and tools are accumulating across teams. The customer experience team is building support agents. The finance team is exposing billing tools. Logistics is wrapping its tracking systems. Operations is automating ticket creation. Each team builds against its own conventions, its own vocabulary, its own sense of what a customer is and what a transaction looks like. Individually, each tool works. The promise of agent ecosystems is that these tools combine into workflows that cross those team boundaries, so that a single intent like "find customers with shipping delays and open billing disputes" can be served by composing tools that no single team owns.

That promise breaks at scale.

The breakage isn't in any individual tool. It sits in the seams between them. When a workflow needs the logistics team's shipment tracker to feed the finance team's billing query, something has to decide that the tracker's customer_ref and the billing system's customer_id are the same thing, and that the type alignment isn't enough on its own. Today, that decision is either hand-wired by a developer who happens to know both systems, or fuzzy-matched at runtime by an LLM with no organizational context. Hand-wiring works at five tools. It collapses at five hundred.

Five hundred is not a hypothetical. Internal API counts at large organizations cross that threshold routinely; agent counts are heading the same direction faster, because every team that built APIs is now building agents on top of them. The composition surface across that scale isn't a single team's coordination problem. It is an organizational problem, and it is shaped by the same forces that make any organizational integration hard: teams that evolved independently, vocabularies that drifted apart, mergers that fused incompatible models, domain experts who left, and the absence of any authoritative answer to questions like "what is a customer." A workflow that crosses three teams crosses three vocabularies. A workflow that crosses three business units in a conglomerate may cross three different definitions of the entities it operates on.

The available standards don't help. OpenAI's function calling, Anthropic's tool format, and the Model Context Protocol all describe what a tool takes as input: name, parameters, types, descriptions. None of them describe what a tool outputs, let alone what its output fields mean. The downstream LLM is expected to figure that out from context. That is tractable for a single tool call. It is not tractable for a graph of tools where outputs feed inputs across team boundaries, semantic conventions, and domain models that nobody agreed on in advance.

The cost of getting this wrong scales with the workflow. A misjoined field in a casual workflow produces a confused user. A misjoined field in a workflow that triggers refunds, dispatches technicians, or sends customer communications produces operational damage that takes weeks to detect and longer to unwind. At organizational scale, silent semantic mismatches are not edge cases. They are the failure mode.

Most organizations don't feel this acutely yet because they're still in the five-tools, single-team, single-vocabulary phase. The architectural decisions made in that phase determine whether the five-hundred-tool, multi-team, heterogeneous-vocabulary phase is solvable or whether the organization spends its agent budget rebuilding the same connective tissue that every other organization is also rebuilding, badly, in parallel. The window to make those decisions well is now, before the substrate ossifies.

This paper describes a system for compositional tool retrieval at organizational scale across heterogeneous agent networks. We call it Ratatoskr.

2. Why Now, Why This Is Hard

Three structural failures show up wherever organizations try to compose tools across teams.

Schema mismatch. Tools built by different teams use different conventions. One team's user_id is another team's account_number is another team's subscriber_ref. The types may align, all strings, all roughly UUIDs, but type alignment doesn't establish meaning.

Semantic drift. Even when field names match, meanings diverge. A telecom organization with M&A history (the Verizon shape) has billing systems where customer_id means the entity being billed, and network operations systems where customer_id means the technical account associated with installed equipment. Both are valid. Both are different. A workflow that silently treats them as interchangeable produces wrong answers in cases nobody catches.

Discoverability. At fifty tools, an architect can hold the catalog in their head. At five hundred, nobody knows what exists. Workflows get hand-built around the tools the author happens to remember. Better tools sit unused. Duplicate tools get built because the existing one wasn't found.

These problems compound. Schema mismatch and semantic drift make the existing tools harder to combine. Discoverability failure makes the right tools invisible. The result is that agent ecosystems get worse as they grow, not better. Past a threshold, adding tools subtracts utility.

The available approaches don't solve any of this:

  • Function calling specs (OpenAI, Anthropic) describe inputs to a single tool. There is no notion of output schema, output semantics, or composition. Each call is independent.
  • MCP improves on this by giving tools a richer registration protocol, but it remains input-shaped. Tools advertise what they accept, not what they emit, and certainly not what their outputs mean.
  • Agent frameworks (LangChain, LangGraph, Autogen) provide orchestration primitives, but assume the developer already knows which tools to use and how their outputs map to downstream inputs. They solve coordination, not discovery or compatibility.
  • Manual orchestration (workflow builders, integration platforms) works for known flows but doesn't scale to dynamic composition based on user intent.

What's missing is a substrate: a layer that captures not just tool signatures but tool semantics, and uses those semantics to make composition decisions tractable.

The same structural pattern shows up across organizational shapes. A diversified industrial conglomerate (the 3M shape) has divisions that evolved independently, each with its own customer model. A manufacturing-and-service business (the Cummins shape) has "customer" meaning the dealer in one context and the end-user in another, both valid, both incompatible if naively merged. A scientific instruments company (the Agilent shape) has domain vocabulary so specialized that ontology depth varies dramatically by sub-domain. None of these are insider observations. They are structural consequences of how organizations grow.

This problem is three to four years away from being acute for most organizations. That is exactly why the architecture matters now. By the time the problem is undeniable, the organizations that built composability into their substrate will be moving while the ones that didn't are rebuilding.

3. System Overview

Ratatoskr is a platform for organizational agent infrastructure. The full vision includes governance, orchestration runtime, security, agent versioning, and several other components. This paper covers one of those: compositional tool retrieval.

Representation Layer captures what tools are. The canonical tool description format records inputs, outputs, and output field semantics. The ontology service resolves field meanings against an organizational concept model. The translator layer maps the canonical format bidirectionally to vendor formats so tools registered in any standard can participate.

Discovery Layer finds tools relevant to an intent. The intent parser converts natural language to a structured query. Semantic retrieval against the tool registry produces a candidate set. The vector store backs this layer.

Reasoning Layer decides how candidate tools combine. The schema-compatibility graph filters candidates by which tools can actually feed each other based on output-input compatibility. The composition planner emits a single plan, with confidence gating its shape. The path ranker picks among multiple candidate paths.

Control Layer enforces correctness boundaries. Human-in-the-loop approval gates any new graph edges proposed by the LLM. Tiered failure modes ensure the system fails honestly rather than emitting plans it can't justify.

This paper covers compositional retrieval. The broader Ratatoskr vision, which includes the governance engine, orchestration runtime, security layer, agent versioning, and observability, is part of Ratatoskr's roadmap but out of scope here.

The intent parser produces a structured representation of the request: what state the workflow starts from, what state it should reach. Retrieval finds candidate tools whose descriptions match the structured query semantically.

The graph layer is where ontology becomes load-bearing. To decide whether one tool's output can feed another tool's input, the graph layer needs to know whether their fields refer to the same concept. It queries the ontology service to resolve field anchors, check class equivalence, and walk class hierarchies. Two fields anchored to the same ontology concept are compatible. Two fields anchored to equivalent concepts (a BillingCustomer and a Customer declared equivalentTo) are compatible. Hierarchy and subclass relationships are also resolved through the ontology, with full treatment in Section 6. The graph layer's compatibility decisions are grounded in these ontology resolutions, not in field name matching or fuzzy text comparison.

When pre-existing edges already establish a compatible path, the graph layer hands the candidate path directly to the composition planner. When edges are missing but both endpoints are ontology-anchored, the graph layer asks the ontology service whether the anchors are equivalent and produces a high-confidence edge proposal if they are. When edges are missing and the fields are not ontology-anchored, the graph layer falls back to LLM-driven proposal based on descriptions and aliases, marked as low-confidence.

The path ranker handles cases where multiple candidate paths exist. The composition planner classifies the situation into one of five tiers based on graph coverage and proposal confidence, and either emits a single plan or routes proposed edges to the human approval queue. Approved edges persist into the graph and become available for future queries.

The plan is emitted, not executed. Execution belongs to the orchestration runtime, which is a separate Ratatoskr component out of scope for this paper.

5. The Tool Description Format

The canonical tool description format is the registration substrate for Ratatoskr. It captures three things:

  • What the tool accepts as input. The standard surface that function calling specs already cover.
  • What the tool emits as output. Currently absent from major specs.
  • What each output field means. The part that makes composition tractable.

The third responsibility is what existing formats miss. Without it, an output schema is just typed data: { shipment_id: string, destination_zip: string, estimated_delivery: string }. A downstream tool that needs an address has no way to know whether destination_zip is the right field to use, whether it's complete, or whether the format conventions match what's expected. With output field semantics, each field carries enough metadata for the graph layer and the LLM to reason about compatibility before runtime.

Three field sources

Every field in the canonical format declares one of three sources:

  • ontology. The field is anchored to a concept defined in the organizational ontology. The format carries a reference to the concept (e.g., Customer.cust_id); descriptions, examples, and authoritative type info come from the ontology service. This is the preferred source.
  • custom. The field has no ontology anchor. The author provides a description, optional aliases, and examples directly in the field definition. This is the fallback for fields not yet covered by the ontology, or for tool-local concepts that don't warrant promotion.
  • operational. The field is system telemetry: execution time, request IDs, trace tokens, rate-limit headers, response timestamps. It's part of the tool's output but irrelevant to composition. The graph layer ignores operational fields when assessing compatibility.

The distinction between custom and operational is not about the field's nature but about its composition relevance. Two tools that both emit executionTimeInMs should not have a graph edge proposed between them on that basis; the field is operational. Conversely, a created_by field might be operational in a customer lookup tool (just metadata) but domain-relevant in a compliance audit tool (the actual subject of the audit). At registration time, the LLM proposes an initial classification based on field names, descriptions, and the tool's stated purpose. The author reviews and confirms or overrides. This matches the human-in-the-loop pattern used throughout the system.

Worked example: a single field across all three sources

Consider a billing tool that emits a customer reference. The same conceptual field looks different depending on its source declaration.

Ontology-anchored:

field: cust_id
source: ontology
anchor: Customer.cust_id
type: string
example: "CUST-88421"
optional: false

The ontology service supplies the description, alias set, and authoritative type. The tool author writes nothing about meaning; the anchor handles it.

Custom (no ontology anchor):

field: cust_id
source: custom
description: "The customer placing the order. For B2B orders, this is the purchasing organization's ID, not the individual user."
aliases: [customer_id, cust_ref, client_id]
type: string
example: "CUST-88421"
optional: false

The author carries the full burden of describing meaning. The field works, but it depends on the author's care and consistency.

Operational:

field: executionTimeInMs
source: operational
description: "Time taken to process the request, in milliseconds."
type: integer
example: 142
optional: true

Type and description for documentation. The graph layer ignores this field entirely.

Registration paths

Tools enter the registry through one of two paths.

Direct canonical registration. The author writes the canonical format directly. They have full control over input schema, output schema, and field semantics. This is the preferred path for new tools.

Vendor format registration. The author submits an existing tool spec in OpenAI's function format, Anthropic's tool format, MCP, or another vendor format. The translator layer extracts the input schema automatically. The output schema and field semantics must be supplied manually by the author, because vendor formats do not describe outputs and the LLM has no signal to fabricate them: a description like "returns customer details" tells nothing about whether the output has five fields or fifty, what types they have, or which concepts they map to in this organization. The author provides this information as part of registration.

In both paths, the registration flow is ontology-aware. The author declares which ontology classes the output is constructed from, and the registration interface offers type-aware anchoring suggestions for fields drawn from those classes' properties. The LLM proposes anchors, types, and domain-versus-operational classification; the author reviews and confirms.

On custom fields

Custom fields exist as a gradual-adoption fallback for fields not yet covered by the ontology. They are not the recommended path. The system makes anchored fields cheap to register and custom fields observably costly: registering a field as custom requires a justification (concept missing from ontology, tool-local computation, uncertain), custom field counts are tracked per tool and per author, and downstream composition over custom fields is slower because every edge requires fresh human approval. Anchored fields produce faster composition, fewer approvals, and higher graph reliability. The architecture rewards discipline rather than enforcing it through hard blocks, because hard blocks would make registration impossible during early ontology adoption when the ontology is still incomplete.

Translator layer (emission direction)

When a plan is generated and a downstream tool needs to be invoked, the translator produces the input payload in whatever vendor format the tool expects. This direction is lossless because the canonical format is a strict superset of vendor input formats. The plans Ratatoskr emits include field-level mappings that can be translated into native vendor calls without ambiguity.

The output side is canonical-only because no vendor format describes outputs. That gap is, in fact, the gap Ratatoskr's format exists to fill.

Scope: JSON-only in V1

The canonical format describes JSON-shaped inputs and outputs. Natural-language outputs are out of scope for V1. This is not a gap to apologize for; it is a deliberate scoping decision. NL-to-JSON or JSON-to-NL handoffs are an extraction problem, distinct from a composition problem. Where NL appears in a workflow, an extraction adapter (an LLM call that produces structured JSON from NL) is itself a tool that can register with Ratatoskr in the canonical format. The composition logic stays JSON-shaped; the NL adapter is just another node in the graph.

6. Ontology as Runtime Substrate

The format alone isn't enough. Author-declared semantics, descriptions, aliases, examples written by the tool author, work for small ecosystems and break at organizational scale, for the same reasons that schema-mismatch and semantic-drift break at scale: every author writes what makes sense to them, in their team's vocabulary, and the result is a patchwork of overlapping and conflicting micro-vocabularies.

The fix is an organizational ontology that defines concepts once, authoritatively, and makes those concepts available to tools at registration time. This isn't a new idea. Ontologies have existed for decades in domains like healthcare, finance, and manufacturing. They've largely failed to be useful, because they had no consumer; they were maintained as documentation projects, not as runtime infrastructure.

Ratatoskr is the consumer that makes the ontology load-bearing. Tools register against ontology concepts. The graph layer trusts ontology-anchored compatibility. The LLM reasons over a structured semantic model rather than fuzzy-matching prose. Ontology stops being a metadata graveyard and becomes the substrate that makes composition tractable.

Ontology is part of Ratatoskr, not external

Expecting organizations to build ontology infrastructure as a precondition for adopting Ratatoskr is a non-starter. The ontology service is therefore part of Ratatoskr itself. It ships with the platform and provides registration, lookup, and equivalence-resolution APIs. Organizations populate it with their concepts; they don't build the substrate. Bootstrap concepts (Customer, Order, Employee, Invoice, and similar near-universal entities) come pre-defined; organizations extend with domain-specific concepts.

What the ontology contains

The ontology contains classes (Customer, Order, Employee, Equipment, Shipment, BillingHistory, and so on), each with data properties (the fields that belong to instances of that class) and object properties (relationships to other classes). Classes can form hierarchies. A B2CCustomer is a Customer, a B2BCustomer is a Customer, and both inherit properties from the parent class.

Two classes can be declared equivalent (Customer in the billing system is the same concept as Customer in the CRM, even though the systems track different attributes). Two classes that look similar can be declared distinct (shipping.Customer representing the recipient of a delivery is not the same concept as billing.Customer, in B2B contexts where the recipient and the billed party can differ). The ontology forces the organization to make these calls explicitly at modeling time, rather than letting them surface as silent bugs at runtime.

Hard cases the ontology handles

Same concept, different field names across teams. Team A's client_id, Team B's customer_ref, Team C's subscriber_id all anchor to Customer.cust_id in the ontology. The graph layer treats them as compatible because the ontology says so.

Different concepts, same name. shipping.Customer and billing.Customer have separate class definitions. A field anchored to one is not compatible with a field anchored to the other, even though both look like "the customer."

Subclass variation. A tool built for B2BCustomer doesn't compose with a tool built for B2CCustomer, but a tool built for the root Customer class composes with both, because subclasses are-a Customer.

Derived concepts. Customer.tenure_years is computed (current_date - Customer.signup_date). If the derivation is organizationally meaningful, where multiple tools across the organization want this concept, it gets promoted to a derived data property on the Customer class. If it's tool-local, it stays a custom field on the tool that needs it.

Query-shape fields. A search tool has input fields like start_date and end_date for filtering BillingHistory records. These don't map directly to properties on BillingHistory. They map to operations on those properties. The format expresses this as a field anchored to BillingHistory.billing_period_start with a role annotation indicating filter semantics (filter:gte).

What the ontology does not handle

Operational fields, things like execution time, request IDs, trace tokens, don't belong in the ontology. They're system telemetry, not domain concepts. Forcing them in produces junk-drawer classes that pollute the ontology and reduce its usefulness. The format keeps operational fields outside the ontology entirely.

Tool-local computed values that no other tool needs also stay outside. Promoting every author's one-off derived field to the ontology defeats the point of having a shared model.

Author-declared as gradual-adoption fallback

Real ontologies are incomplete. They start small and grow as the organization formalizes its concepts. Ratatoskr accommodates this through the custom field source: fields not yet covered by the ontology can be registered with author-declared semantics. Composition over custom fields is heuristic. The LLM proposes edges based on prose matching, and every such edge requires fresh human approval, with no fast-path even for previously-seen patterns. This is part of the cost gradient described in Section 5: custom fields are allowed, tracked, and observably more expensive to compose with than anchored fields.

Over time, recurring custom concepts get promoted to the ontology, the corresponding fields get re-anchored, and the heuristic edges become deterministic ones. The format works across the gradient from "ontology covers nothing" to "ontology covers everything," and the organization migrates at its own pace.

This dual-mode design is part of what makes the system adoptable. Organizations don't need a complete ontology to start; they get value immediately and improve the substrate over time.

7. Storage and Retrieval

Tool descriptions registered in the canonical format are stored in a vector database for semantic retrieval. V1 uses Chroma. The storage interface is designed so the vector store is pluggable; no Chroma-specific assumptions are baked into the retrieval logic.

Retrieval proceeds in three steps:

  • Semantic similarity over the registered tool descriptions, given the structured query from the intent parser. This produces a shortlist of candidate tools whose descriptions are semantically close to the intent.
  • Graph reachability filter. The candidate set is narrowed to tools that can actually participate in a composable path, meaning their inputs and outputs connect to other relevant tools through edges in the schema-compatibility graph (Section 8).
  • Path ranking. Where multiple candidate paths exist, the ranker chooses one.

The V1 path ranker is deliberately basic: shortest path wins, with ties broken by the proportion of edges in the path that are graph-backed (pre-existing, high-confidence) versus LLM-proposed (newer, lower-confidence). Sophisticated ranking, things like success-rate-weighted, cost-aware, multi-criteria, is parked for future work.

For evaluation, V1 uses a synthetic test set of 50 agents anchored on real public API specifications across 5 to 6 domains (telecom, customer/CRM, finance, communications, analytics). Ten of these are deliberately constructed as dangling nodes: tools whose inputs cannot be satisfied by any other tool's output in the registry. Dangling tools test the system's ability to retrieve based on semantic match while correctly refusing to compose.

8. Graph Layer and Composition Planner

The graph layer holds the schema-compatibility relationships between tools. V1 uses Neo4j.

Nodes are tools. Edges represent the assertion that one tool's output can satisfy another tool's input, meaning both type alignment (the underlying types match) and semantic alignment (the fields refer to the same concept, by ontology anchor or by approved alias). An edge is a strong claim: it says these two tools can be composed without further reconciliation.

Edge provenance

Every edge in the graph carries provenance metadata indicating how it was created:

  • Human-curated. A platform engineer or domain expert added the edge directly, based on knowledge of the tools.
  • LLM-proposed, human-approved. An LLM analyzed two tool descriptions, proposed that a compatibility relationship exists, and a human reviewed and approved the proposal before it was persisted.
  • Human-created-manually. Similar to human-curated but added through the registration UI rather than direct graph edits, typically by the tool author at registration time.

Each edge also carries an audit history: who created it, when, what the LLM's confidence was if applicable, and any subsequent modifications.

The five-tier composition planner

The composition planner is the decision point that turns retrieval and graph data into a plan. It classifies every query into one of five tiers based on graph coverage of the candidate path and LLM confidence about any gaps.

Tier 1: Full graph coverage, high confidence. Every edge in the candidate path already exists in the graph and is well-established. The plan is emitted directly. This is the steady-state happy path.

Tier 2: Partial graph coverage, high LLM confidence on the gap. Most edges exist; one or two are missing. The LLM proposes the missing edges with high confidence (both endpoints are ontology-anchored, or the LLM has strong evidence from the descriptions). The plan is emitted, but the proposed edges enter the human approval queue for review.

Tier 3: Partial graph coverage, low LLM confidence. Edges are missing and the LLM is uncertain about them. The plan does not auto-emit. The proposed edges go to human approval, and the plan is held until they are either approved or rejected.

Tier 4: No graph coverage, but LLM proposes a high-confidence path. None of the needed edges exist, but the LLM has strong evidence that a viable path exists. Same handling as Tier 3: approval required before emission.

Tier 5: No path, low confidence. The system cannot construct a viable plan and has no high-confidence proposals. The query fails honestly with a diagnostic indicating what's missing, typically tools or edges that would close the gap.

The planner emits one plan per query, not multiple options for the user to choose between. When multiple candidate paths exist within the same tier, the path ranker selects one. This is a deliberate design choice: presenting multiple plans pushes the disambiguation cost onto the user, who is least equipped to make it. Ratatoskr is responsible for picking the best plan it can defend, or failing honestly when it cannot.

Confidence gates the shape of the plan, not its presence. A high-confidence plan emits with full structure. A lower-confidence plan emits with explicit annotations about which edges are LLM-proposed and pending approval. The user always knows what tier they are operating in.

9. Human Roles in Composition

Two distinct human roles exist in Ratatoskr's composition flow. Both are deliberate, and neither is provisional.

Human as workflow initiator

Every composition is human-triggered. Ratatoskr does not propose workflows on its own. Tools are registered, the graph self-extends through approved proposals, but no workflow is composed until a human expresses an intent. This is a design choice, not a limitation. Autonomous workflow generation, where agents decide on their own to compose other agents, is outside the trust envelope of organizational deployment.

Human as edge approver

Every LLM-proposed edge requires human approval before being persisted to the graph. The LLM analyzes tool descriptions and proposes that an output of one tool is compatible with an input of another. The human reviews the proposal, the rationale, the confidence score, and either approves or rejects. Approved edges enter the graph with LLM-proposed-human-approved provenance and are subsequently used as if they were human-curated.

This design accepts a real cost: human approval scales linearly with the rate of new tool registrations and intent queries. At low volume, this is invisible. At high volume, in large organizations with hundreds of tools, the approval queue becomes a bottleneck.

V1 accepts the bottleneck because the alternative is worse. LLMs hallucinate, and edges persist; a confidently-wrong edge silently corrupts every workflow that uses it. The auditability claim, that every edge in the graph has known provenance and a human approver, is foundational to enterprise trust, and it doesn't survive shortcuts.

V2 work focuses on making approval scale, not removing humans from the loop. Batch approval flows let reviewers handle dozens of proposals at once. Domain-delegated approvers route telecom edges to telecom team reviewers, finance edges to finance reviewers. Tiered approval distinguishes high-stakes edges (cross-domain, high-impact tools) from low-stakes ones (clearly ontology-anchored, narrow impact). Smarter pre-validation auto-classifies proposals where both endpoints are ontology-anchored as low-risk, accelerating reviewer attention.

Confidence-gated auto-approval, where high-confidence ontology-anchored proposals bypass human review, is part of the longer arc. It becomes plausible when verification mechanisms exist to catch the wrong-but-confident case, and when the organizational ontology has matured enough that anchoring is the norm. It is not in scope for V1 or V2. The system's correctness story does not depend on auto-approval existing.

10. Failure Modes

Ratatoskr's behavior under uncertainty is governed by a single principle: prefer honest failure over confident-wrong output.

Most agent systems default to producing something. A user asks for a workflow; the system returns a workflow. If the system isn't sure, it returns its best guess and lets the user discover the problem at runtime. This is reasonable for casual tooling and unreasonable for organizational infrastructure, where wrong workflows produce wrong actions on real systems.

Ratatoskr defaults to producing only what it can defend. The five-tier planner is the mechanism. Each tier has a decreasing trust gradient, and the system always knows which tier it is in. Tier 1 emits without ceremony. Tier 5 refuses to emit a plan and instead returns a diagnostic.

Worked failure scenario

A user expresses an intent: "Find all customers in the past quarter who had both a shipping delay and an open billing dispute, and route them to a senior support agent."

The intent parser produces a structured query. Retrieval finds candidate tools: a customer search tool, a shipment status tool, a billing dispute lookup, and a support routing tool. The graph layer attempts to construct a path. The customer search and the billing dispute tool have a graph-backed edge. The shipment status tool has not been previously composed with the billing dispute tool; the LLM is asked to propose an edge. The proposal comes back with low confidence, because the relevant fields aren't ontology-anchored, the descriptions are brief, and the LLM cannot confidently assert that the shipment's customer_ref is the same concept as the billing dispute's customer_id.

The composition planner classifies this as Tier 5: critical edges are missing, LLM confidence is low, and no plan can be defended. The system does not emit a plan. It returns a diagnostic that surfaces what was discovered, where composition broke, and what the system suggests for closing the gaps.

  - Billing Dispute Lookup
  - Support Routing

Composition gaps:

  1. Shipment Status → Billing Dispute Lookup
     No compatible edge. The customer_ref field on Shipment Status and the
     customer_id field on Billing Dispute Lookup are not anchored to a
     shared ontology concept.

     Suggestions:
       - Anchor both fields to Customer.cust_id in the organizational
         ontology. (confidence: high)
       - Register a bridging tool that joins shipment and billing records
         by customer reference. (confidence: medium)

  2. Billing Dispute Lookup → Support Routing
     Edge exists in the graph but expects a priority field that Billing
     Dispute Lookup does not currently emit.

     Suggestions:
       - Extend Billing Dispute Lookup to emit a priority field anchored
         to SupportTicket.priority. (confidence: high)
       - Insert a priority-inference tool between the two. (confidence: low)

The user gets nothing executable. They also get a clear inventory of what was discovered, where composition broke, and what changes would close the gaps. Confidence scores on each suggestion let the user prioritize: high-confidence suggestions are likely correct and need light review, low-confidence suggestions need scrutiny before acting. The system has failed visibly, not silently, and it has failed usefully.

Other failure modes

Multiple candidate paths. The ranker emits one. The single-plan-per-query principle holds because presenting alternatives pushes the disambiguation cost onto the user, who is least equipped to make it. Other paths are suppressed.

Schema reconciliation needed. A workflow requires that one tool's output field be transformed before it can serve as another tool's input field (aggregating, computing, splitting). V1 does not handle field-level reconciliation. Such cases fail with an explicit diagnostic, rather than silently emitting a plan with implicit lossy mapping.

Partial graph coverage. Handled by the planner tiers. The user sees that some edges in the plan are LLM-proposed pending approval, and can choose to wait for approval or amend the intent.

The common thread is visibility. The system exposes its uncertainty; it does not hide it.

11. Design Decisions and Limits

Trade-offs taken

JSON-only handoffs in V1. Composition reasons over structured data. NL handoffs are out of scope; where they appear in workflows, an extraction adapter is itself a tool that participates in composition.

Single plan per query. No multi-plan exploration. The system either commits to a defensible plan or fails. Multi-plan exploration is a UX feature that pushes disambiguation cost to the user, which is the opposite of what the system should do.

Human-in-the-loop for all LLM-proposed edges. Correctness over throughput. The bottleneck is acknowledged and addressed in V2 through approval ergonomics.

Basic path ranker. Shortest path with graph-backed-edge tie-breaking. Sophisticated ranking, things like success-rate-weighted, cost-aware, multi-criteria, is parked.

Ratatoskr ships ontology infrastructure. Organizations populate the ontology with their concepts; they don't build the substrate. This makes the ontology layer a Ratatoskr-internal component, not a third-party dependency.

Parked for future work

  • Field-level schema reconciliation (transformations, aggregations, splits between output and input fields)
  • Edge revocation flows (graph maintenance when tools change or compatibility assumptions are invalidated)
  • Sophisticated path ranking
  • Sophisticated derivation expression for derived ontology concepts
  • Ontology evolution and versioning at scale
  • NL handoff support

Out of scope for compositional retrieval

Several capabilities belong to other Ratatoskr components or external systems:

  • Governance. Policy enforcement, access control over which tools can compose with which, role-based composition restrictions.
  • Orchestration runtime. Actual execution of the plans Ratatoskr emits.
  • Security and authentication. Credential management, agent identity.
  • Agent versioning and lifecycle. Handling tool updates, deprecation, backward compatibility.
  • Registration drift. Tool registrations are assumed accurate. Detecting drift between registered descriptions and actual tool behavior, whether from code changes, API updates, or semantic shifts, falls under the agent versioning and lifecycle management component.
  • Observability. Runtime telemetry, workflow tracing, performance monitoring.

These are part of Ratatoskr's broader scope. They are not part of the compositional retrieval component described in this paper.

Timing

This problem is early. Ratatoskr's design optimizes for the long arc, not immediate adoption. There is a possibility the industry doesn't move toward compositional agent infrastructure at all. Given the current trajectory of agent tooling proliferation, that scenario looks unlikely. The organizations that take the substrate seriously now will be moving while the ones that don't are rebuilding.

12. Future Work

Future work splits into two parts: extensions within the compositional retrieval component, and development of the broader Ratatoskr platform.

Within compositional retrieval

  • Sophisticated path ranking driven by historical success rates, cost models, and multi-criteria optimization.
  • Field-level schema reconciliation, including the language for expressing transformations between output and input fields.
  • Natural-language handoff support via extraction adapters that participate as composable tools.
  • Edge revocation and graph maintenance flows for handling tool updates and compatibility regressions.
  • Approval ergonomics: batch approval, domain-delegated reviewers, tiered approval for low-risk versus high-risk edges, smart pre-validation. Eventually, confidence-gated auto-approval for ontology-anchored edges with verification mechanisms.
  • Ontology layer expansion: richer reasoning beyond flat concept lookup, ontology evolution and versioning, federated ontologies across business units.

Across Ratatoskr as a platform

  • Governance engine. Policies governing which tools can compose with which, access controls, audit and compliance for composed workflows.
  • Orchestration runtime. Execution of Ratatoskr-emitted plans, with state tracking, retry handling, and error recovery.
  • Security and authentication layer. Credential management for composed workflows, agent identity, secret handling.
  • Agent versioning and lifecycle management. Semantic versioning of tools, deprecation flows, backward compatibility guarantees, and detection of drift between registered tool descriptions and actual tool behavior.
  • Observability and telemetry. Runtime instrumentation, workflow tracing, performance and reliability monitoring across composed agent paths.

The compositional retrieval component is the foundation. The other components extend Ratatoskr's scope and will be developed in subsequent work.

Ratatoskr is named after the squirrel that ran messages between the realms of Yggdrasil in Norse mythology. The system's job is the same: move information between tools intentionally, without distortion or harm.
